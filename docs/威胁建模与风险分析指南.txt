1概述
1.1目的
本指南旨在规范化产品的安全评估流程，通过系统化的威胁建模方法，识别产品的潜在风险，并指导制定科学的消减策略。
1.2适用范围
本指南适用于统信软件研发的操作系统、应用软件、管理平台等产品的设计。面向的读者包括产品对应的产品经理、研发人员、测试人员。
2建模指南
安全工程师或架构师应基于建模模型和风险评估模型，按照建模流程对威胁进行量化分析，输出威胁建模与风险分析报告。
2.1建模模型
本指南采用微软的 STRIDE 模型进行威胁建模，并结合风险矩阵 进行量化分析。
建模时应从以下六个维度进行威胁分析：
表 1STRIDE模型
威胁维度	定义与攻击场景	消减措施
S - Spoofing
(仿冒)	攻击者冒充他人身份（如冒充 root、冒充客户端进程）。
例：利用弱密码 SSH 爆破；IP/ARP 欺骗。	身份认证强校验：
1. 禁用密码登录，仅允许 SSH 密钥对。
2. Socket 通信校验 SO_PEERCRED (获取对端 UID/PID)。
3. 启用 TLS 双向认证 (mTLS)。
T - Tampering
(篡改)	未经授权修改数据或代码。
例：修改 /etc/shadow；植入恶意内核模块；更改二进制文件。	完整性保护：
1. 文件权限严格控制 (如 chmod 644, root only)。
2. 配置FileArmor配置访问策略。
R - Repudiation
(抵赖)	攻击者否认其恶意行为，系统缺乏证据。
例：删除 bash_history；停止日志服务。	审计与日志：
1. 配置 auditd 监控关键系统调用 (execve, open)。
2. 日志远程投递 (Rsyslog) 防止本地擦除。
3. 确保日志包含时间戳和操作源 IP/ID。
4. 配置FileArmor配置访问策略。
I - Information Disclosure
(信息泄露)	未经授权读取敏感信息。
例：/proc 泄露内核地址；日志打印明文密码；内存未擦除。	加密与权限隔离：
1. 启用全盘加密或分区加密保护敏感数据。
2. 严格的目录权限 (避免 777)。
3. 进程隔离 (玲珑) 防止跨进程读取。
D - Denial of Service
(拒绝服务)	破坏服务的可用性。

例：Fork 炸弹耗尽 PID；SYN Flood；磁盘写满。	资源限制：
1. 使用 Cgroups 限制 CPU/内存配额。
2. 配置 ulimit 限制文件句柄数和进程数。
3. 网络层配置 iptables 限流。
E - Elevation of Privilege
(权限提升)	从低权限账户获取高权限 (通常是 root)。
例：利用 SUID 程序漏洞；内核提权漏洞。	最小权限原则：
1. 移除不必要的 SUID/SGID 位。
2. 使用 Linux Capabilities (如仅给 CAP_NET_ADMIN) 代替完整 root。
3. 启用 Seccomp 过滤系统调用。

2.3建模流程
(1)资产识别：根据产品的架构图或功能结构图，定义信任边界，确定核心资产，绘制数据流图。
(2)威胁分析：按照STRIDE模型分析核心资产潜在的风险。
(3)风险评估：采用风险评估模型对风险进行评级。
(4)消减措施：为每种风险制定对应的消减措施。
2.3.1资产识别
产品在设计阶段，需求和设计方案并未完成，此阶段仅有产品目标或原始需求，架构师和产品经理需要根据这些信息来进行整体架构设计或功能结构设计。完成后，架构师或研发代表则需要与安全工程师一起根据产品目标、原始需求、架构设计或功能结构设计识别出可用于数据传递相关的对象，这些对象即是需要保护的资产。
资产可按照下述方法进行识别：
找入口：即产品有哪些途径可接收外部输入。如：DBus 接口、CLI 参数、监听的 Socket、读取的配置文件等。
找处理：即产品如何处理这些数据。如以 root 运行的主守护进程、fork 出的子进程、调用的 shell 脚本、调用其他进程的接口等。
找存储：即产品如何存放经过处理的数据。如写入本地文件、写入数据库、写入内存等。
找依赖：即产品需要调用哪些其他组件来实现功能。如依赖 systemd 启动、调用 iptables 命令、链接 openssl 库等。

资产通常包括：
数据：配置文件、密钥、用户数据、日志。
进程：守护进程、临时任务、API 接口。
通信：Socket 文件、网络端口、共享内存。

完成资产识别后，接下来可以按以下方法确定信任边界：
身份与权限是否发生变化：数据处理的进程UID/GID是否变化。如数据从普通用户进程流向了特权用户进程；数据从容器流向了宿主机；进程内进行了用户切换（提权到root）等。
数据来源是否受产品控制：读取、接收的数据是否完全受本产品控制。如数据来源于用户输入、环境变量、进程间通信接口(socket文件、DBus接口等)、远程通信协议(HTTP、Websocket等)、磁盘上的文件等，这便跨越了数据边界。
网络与物理位置是否发生变化：数据是否会跨网络传输。如上报数据到更新管理平台、日志采集平台，或备份文件到云端等。

资产和信任边界确定后，便可进行数据流图的绘制了。在绘图时，使用红色虚线框住具有相同信任级别的组件，任何穿过红线的箭头（数据流），都是需要重点防御的地方。

2.3.2威胁分析
识别资产后，针对每一个资产，逐一匹配 STRIDE 的六个维度。如果该维度不适用（例如“日志文件”通常不涉及“欺骗”），则标记为 N/A。
分析口诀：
对着“数据/文件”问：能被偷看吗(I)？能被改坏吗(T)？
对着“进程”问：能被提权吗(E)？能被搞崩吗(D)？
对着“接口”问：能被冒充连接吗(S)？能被赖账吗(R)？

3附录
3.1常见威胁及消减措施
3.1.1Spoofing（仿冒）
定义：攻击者冒充他人身份（如冒充管理员、系统进程或网络实体）。
常见威胁：
非授权 SSH 访问。
弱认证机制导致账户被破解。
UID/GID 劫持。
ARP 或 IP 欺骗。
消减措施：
身份与访问控制：利用 sudoers 精确授权，禁止直接使用 root 用户登录。
认证强化：增加密码复杂度，启用多因子认证；同时结合“最小权限”原则，确保仅授权用户可进行操作。
3.1.2Tampering（篡改）
定义：未经授权修改数据、代码或配置。
常见威胁：
恶意修改 /etc/shadow或 /usr/bin 下的二进制文件。
篡改内核参数（通过 sysctl）。
修改 systemd 单元文件以持久化恶意服务。
消减措施：
强制访问控制：配置并启用 USEC，实行“默认拒绝”策略，防止非授权进程修改关键文件。
文件完整性监控：配置文件保护，实时监控关键文件指纹，一旦被篡改立即告警；或启用签名验证，拒绝验证识别的文件。
3.1.3Repudiation（抵赖）
定义：攻击者在执行恶意操作后，否认其行为，且系统缺乏证据。
常见威胁：
擦除 /var/log/auth.log 等关键日志文件。
禁用 auditd 进程以隐藏攻击痕迹。
修改文件时间戳以混淆取证。
消减措施：
深度审计：配置 auditd 监控 execve（执行命令）、connect（网络连接）等高危系统调用。
日志外发：日志应通过 TLS 协议实时发送至远程日志中心（如 Syslog-ng），防止本地日志被攻击者篡改或删除。
3.1.4Information Disclosure（信息泄露）
定义：未经授权读取敏感信息。
常见威胁：
/proc 接口泄露内核地址（可能辅助其他攻击）。
不当的目录权限（如设置为 777）导致文件被任意读取。
敏感信息（如密钥、密码）被误写入环境变量或日志中。
消减措施：
静态加密：启用全盘加密或分区加密，防止物理盗窃导致的数据泄露。
内核地址保护：启用 KASLR（内核地址空间布局随机化），增加利用内核信息泄露漏洞的难度。
3.1.5Denial of Service（拒绝服务）
定义：通过耗尽资源使合法用户无法访问服务。
常见威胁：
利用 Fork 炸弹耗尽系统 PID 资源。
磁盘写满攻击。
利用内核漏洞引发 Kernel Panic（系统崩溃）。
SYN 洪水攻击。
消减措施：
攻击面缩减：禁用不必要的内核模块（通过 /etc/modprobe.d/），减少可被利用导致崩溃的代码面。
关闭未使用的端口：关闭未使用的网络端口，减少网络层拒绝服务的入口。
3.1.6Elevation of Privilege（权限提升）
定义：低权限攻击者获取了本不应拥有的高权限（通常是 root）。
常见威胁：
滥用配置不当的 SUID/GUID 权限文件。
利用内核 UAF (Use-After-Free) 或 Double-Free 漏洞。
非法获取 CAP_SYS_ADMIN 等高危能力。
利用 sudo 配置错误提权。
消减措施：
最小权限：移除不必要的 SUID 位文件。
内存保护：确保启用堆栈溢出保护及控制流完整性（CFI），防止缓冲区溢出类漏洞被利用来提权。
系统调用过滤：生产环境应用应通过 Seccomp 限制仅能调用必要的系统函数，减少内核漏洞的攻击面。
